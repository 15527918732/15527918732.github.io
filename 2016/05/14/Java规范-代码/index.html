Java规范-代码

  • Java 重构 规范

1.避免使用复杂表达式条件与分支

现象:if/else嵌套过多，降低代码可维护性

方案:代码分段拆分复杂表达式（method或逻辑运算符合并），使用卫语句（多个return出口），使用多态

2.使用泛型代替原生类型

现象：数据类型（原生类型，引用）,List<String> 参数化类型，List原生类型，原生类型降低程序可读性。参数化类型提供了编

译前检查方便，不用显示的cast，自动完成

List<String> strings = new ArrayList<String>();
//方法add1和add2显示了List和List<Object>的区别
//方法add1
//由于参数是List，当把List<String>作为参数传进来时，list添加了一个Integer类型，
//string列表中添加了Integer类型的元素，破坏了安全性
add1(strings,new Integer(1));
String s = strings.get(0); //出错，因为是Integer类型

//方法add2无法通过编译，因为无法接受List<String>作为参数
//The method add2(List<Object>, Object) in the type ListDemo is
//not applicable for the arguments (List<String>, Integer)
add2(strings,new Integer(1));

//方法add1和add3显示了List和List<?>的区别
//方法add3无法通过编译，List<?>一般只能用于读取，不能add，除非是null
//The method add(capture#1-of ?) in the type List<capture#1-of ?>
//is not applicable for the arguments (Object)
add3(strings,new Integer(1));

泛型是在编译器实现的，而不是在虚拟机中实现，虚拟机对泛型一无所知。所以，编译器一定要把泛型类修改为普通类，才能够在虚

拟机执行。类型擦除的关键在于从泛型类型中清除类型参数的相关信息，并且再必要的时候添加类型检查和类型转换的方法。在使用

泛型时，任何具体的类型都被擦除，唯一知道的是你在使用一个对象。比如：List和List在运行事实上是相同

的类型。他们都被擦除成他们的原生类型，即List

3.使用常量或枚举代替魔法数字

现象：魔法数字（Magic Number）是指拥有特殊意义，却又不能明确表现出这种意义的数字。程序里面存在大量这种难以阅读的数字。

方案:定义常量或者常量类或枚举类

4.表达式常量在前

现象：代码存在string.equal("abc")，需要多判断null

方案："abc".equal(string) 会避免NULL判断

5.检查参数有效性

现象：方法有多个参数，内部会对其处理，如果不处理，内部会发生异常

方案：对所需参数逐一校验，为空，正则表达式，类型校验等，数据过滤正常，参与逻辑处理
