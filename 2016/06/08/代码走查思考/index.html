代码走查思考

1.追求新版本特效，导致代码处理显得冗余，效率不一定高。

Optional.ofNullable(obj).isPresent();
if(obj==null)

2.类中方法,如果不需要暴漏外部，就需要定义为private，为了便于后期用户维护，
方法注释必不可少。

3.instance of少用,很影响性能,可以转换成字符串，截取字符串进行相应的业务处理。

4.Controller 中的方法一般一个service，我曾经过度设计，因为查询条件的不同，设计两个子类，导致类的庞大，同时没有添加注释，导致阅读很困难。

5.业务如果不复杂的化，不需要产生额外的类，注意将通用方法，放到工具类。

6.业务稍微复杂些的，想好业务场景，适度抽象，尽量将相似的结构抽成父类，可变的部分
抽象成接口。

7,如果代码出现大量的字符串，而且还有内部拼接，

StringBuffer buf=new StringBuffer(1000);

bff.append(“abc ”);

bff.append(a);

采取String.format(“abc %s”,a);

8,曾经为了将处理不同逻辑分开，将for循环写了两遍，虽然for遍历很快，但是显得很累赘，想想最后还是要合并在一起

For(a){
Do(a);
}

For(a){
Do1(a);
}
合并后的结构:

For(a){
Do(a);
Do1(a);
}

9，服从规范，在特定的地方创建特定的类，也许先前有部分写错啦，不能按照错的进行下去，及时更正。(存在多个项目，不同项目的不同包处理不同的逻辑，遵从最终的规范)

10.不要太相应网上的答案，曾经做个mongodb，mapreduce，它说参数是可选的，其实是有问题，参数必填的，不知是版本问题还是其他什么问题。
